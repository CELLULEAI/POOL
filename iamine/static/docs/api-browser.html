DIRECT ACCESS

Call Cellule.ai from a browser or phone

You don't need a backend proxy. A browser, a PWA or a phone can call the Cellule.ai OpenAI-compatible API directly — CORS is open and the token rides in the Authorization: Bearer header. This is the phone-proxy pattern: the device talks to the pool, the token stays on the device.

Why it works

WhatValue
Base URLhttps://cellule.ai/v1
EndpointPOST /v1/chat/completions
AuthAuthorization: Bearer acc_xxxxxxxx (per-user key, header, never a cookie)
CORSAccess-Control-Allow-Origin: * on every response (incl. the preflight), Authorization + Content-Type allowed, preflight cached 10 min
Native vs web: a native app (Swift / Kotlin / React-Native native) isn't subject to CORS at all — it calls direct, full stop. CORS only matters for a web / PWA / WebView client, and it's already open. Either way: nothing to enable server-side.

Minimal snippet (fetch + Bearer)

// phone-proxy: the device calls cellule.ai directly, token kept on-device.
async function celluleChat(prompt, token, { model = "iamine/raw", signal } = {}) {
  const res = await fetch("https://cellule.ai/v1/chat/completions", {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
      "Authorization": `Bearer ${token}`,
    },
    body: JSON.stringify({
      model,                                  // "iamine/raw" = stateless | "iamine" = memory/conv
      messages: [{ role: "user", content: prompt }],
      stream: false,
    }),
    signal,                                   // optional: AbortController to cancel
  });
  if (!res.ok) throw new Error(`cellule.ai ${res.status}: ${await res.text()}`);
  const data = await res.json();
  return data.choices[0].message.content;     // → "391"
}

Streaming (optional)

Set stream: true to receive a text/event-stream of OpenAI-style data: {...} chunks for progressive display:

const res = await fetch("https://cellule.ai/v1/chat/completions", {
  method: "POST",
  headers: { "Content-Type": "application/json", "Authorization": `Bearer ${token}` },
  body: JSON.stringify({ model: "iamine/raw", messages, stream: true }),
});
const reader = res.body.getReader();
const dec = new TextDecoder();
for (;;) {
  const { value, done } = await reader.read();
  if (done) break;
  for (const line of dec.decode(value).split("\n")) {
    if (!line.startsWith("data: ")) continue;
    const payload = line.slice(6);
    if (payload === "[DONE]") break;
    const delta = JSON.parse(payload).choices?.[0]?.delta?.content;
    if (delta) appendToUI(delta);
  }
}

Security — the token

  • Per-user key (acc_xxxxxxxx) in the Authorization header. Never a master/admin token.
  • Header, not cookie: ACAO: * is incompatible with credentialed cookies, so a Bearer header is exactly right.
  • Store it on-device (iOS Keychain / Android Keystore / secure storage). A token shipped hard-coded in a public bundle is extractable.
  • Revocable: keys can be invalidated server-side at any time, so a leaked key is cheap to rotate.
  • No key yet? Guests get 20 free requests per IP to try the flow; production needs a per-user key. Back off on HTTP 429.

Appeler Cellule.ai depuis un navigateur ou un téléphone

Pas besoin de backend intermédiaire. Un navigateur, une PWA ou un téléphone peuvent appeler l'API OpenAI-compatible de Cellule.ai en direct — le CORS est ouvert et le token voyage dans l'en-tête Authorization: Bearer. C'est le pattern phone-proxy : l'appareil parle au pool, le token reste sur l'appareil.

Pourquoi ça marche

QuoiValeur
Base URLhttps://cellule.ai/v1
EndpointPOST /v1/chat/completions
AuthAuthorization: Bearer acc_xxxxxxxx (clé par-utilisateur, en-tête, jamais un cookie)
CORSAccess-Control-Allow-Origin: * sur chaque réponse (préflight inclus), Authorization + Content-Type autorisés, préflight caché 10 min

Sécurité — le token

  • Clé par-utilisateur (acc_xxxxxxxx) dans l'en-tête Authorization. Jamais un token master/admin.
  • En-tête, pas cookie : ACAO: * est incompatible avec les cookies à credentials — un en-tête Bearer est donc exactement le bon choix.
  • Stocke-le sur l'appareil (Keychain iOS / Keystore Android / stockage sécurisé). Une clé écrite en dur dans un bundle public est extractible.
  • Révocable : une clé peut être invalidée côté serveur à tout moment — une clé fuitée se remplace sans coût.
  • Pas encore de clé ? Les invités ont 20 requêtes gratuites par IP pour tester le flux ; en production il faut une clé par-utilisateur. Recule sur un HTTP 429.
